# Plan: rules installer v2

Wire the downstream rule installer into `/jig init` — the v2 seam left open by
`jig::task::rules-framework-v1`.

## Phase 1 — Installer script · DONE
`templates/rust/rules/install-rules.sh <project-dir>`:
- Preconditions: `kinora` on PATH, target has `.kinora/`, target is a git repo
  with `user.name` set (kinora needs an author). Clear early failures.
- Auto-declares the `rules` root in the target `config.styx` if absent.
- Parses `manifest.toml` (portable awk, no gawk-only features) and stores each
  rule into the `rules` root with its `rule::*` ladder metadata.
- Idempotent: skips rules already resolvable in the target ledger.
- Drift check: manifest-listed source missing → error (non-zero exit); a rule
  `.md` present but unlisted → soft warn (exit stays 0).
- Stages only; prints the `kinora commit` (main) + `git` next steps. Never
  auto-commits (respects kinora's stage-on-branch / commit-on-main discipline).

## Phase 2 — Skill wiring · DONE
`skills/jig/SKILL.md`: `/jig init rust` now installs the rules when kinora is
active (and states that the enforcement layer installs regardless); `/jig sync
rust` re-runs the idempotent installer to pick up newly-added rules.
`manifest.toml` header rewritten to point at the script.

## Phase 3 — Validation · DONE
Tested end-to-end with the rebuilt kinora CLI against throwaway projects: fresh
install (root auto-declared, 3 rules stored), idempotent re-run (all skipped),
orphan-file soft warn, missing-source hard error, kinora-absent / no-.kinora /
no-git-author / no-arg guards, and a full install → `kinora commit` → `git
commit` → resolve-from-committed round-trip.

## Out of scope (still open)
- The generated rung-0 hint digest (query `rule::rung=0`) — not yet automated.
- `--update` mode to re-version drifted rules already in a downstream ledger
  (install currently skips existing; refresh is manual).
- A dylint scaffold.
