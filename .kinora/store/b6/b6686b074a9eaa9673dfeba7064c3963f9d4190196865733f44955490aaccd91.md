# Plan: installer --update mode

Add an `--update` mode to `install-rules.sh` so a project can refresh rules whose
upstream source text changed — the "install currently skips existing" seam left
open by `jig::task::rules-installer-v2`.

## Phase 1 — Implementation · DONE
`install-rules.sh [--update] <project-dir>`:
- Order-independent flag parse; plain-install path unchanged (still skips
  existing rules).
- `ledger_lookup <name>` reads the committed `rules` root styxl to get a kino's
  `id` + current head (same isolated coupling as the digest generator).
- For a present rule under `--update`: compare source body to the resolved body
  (trailing-newline-insensitive). Unchanged → left alone. Changed → re-version
  with `--id`/`--parents` and **no `--root`** (the ambiguous-assign gotcha),
  re-applying current manifest metadata.
- New counters: `updated`, `unchanged`, `deferred`.

## Phase 2 — Docs · DONE
`install-rules.sh` header, `manifest.toml`, and `/jig sync rust` in SKILL.md all
document `--update`.

## Phase 3 — Review + validation · DONE
Fresh-eyes subagent review confirmed `ledger_lookup` offsets, the body compare,
the empty-lookup split, and the re-version invocation are correct; it flagged one
issue — the "changed but not yet committed" case was miscategorized as a hard
`unresolved` failure (exit 1). Fixed: it now increments a soft `deferred` counter
and exits 0 with a "commit then retry" hint.
Verified: no-op when unchanged; re-versions only the changed rule; `kinora commit`
after update is NOT ambiguous; new body resolves; `deferred` path exits 0; plain
install still skips.

## Out of scope (still open)
- Metadata-only graduation (kinora rejects identical-byte re-versions, so a rung
  change needs a body edit too) — acceptable, documented.
- A kinora list/query API to replace the root-blob parsing.
