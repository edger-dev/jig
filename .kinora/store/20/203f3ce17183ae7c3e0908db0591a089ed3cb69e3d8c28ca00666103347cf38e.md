# Rule namespace convention

Every rule kino is named with a **double-colon namespace**:

```
<owner>::<jig-or-domain>::<local-slug>
```

Examples: `jig::rust::no-unwrap-in-lib`, `jig::rust::prefer-file-modules`,
`jig::meta::rule-schema`. The **namespace** is everything before the last `::`;
the **local name** is the final segment. Segments use kebab-case; hierarchy uses
`::`. Owner `jig` is reserved for rules this project ships; third parties use
their own owner segment (`acme::security::...`).

**Why `::`:** it is kinora's *own* `NAMESPACE_SEPARATOR`. A name like
`jig::rust::no-unwrap-in-lib` is therefore already namespaced in kinora's native
scheme — `is_namespaced` / `validate_namespaced` parse it today. When kinora
grows first-class namespace support for names/ids (planned), these ids map onto
it with zero translation: split on the last `::`, the left becomes the native
namespace, the right the local name. This convention is a *bridge* that was
built to disappear cleanly.

**Machine-readable mirror.** Until name-namespacing is first-class, each rule
also carries `rule::ns=<namespace>` metadata so queries and the eventual
migration never depend on string-splitting a title. (Metadata keys are
themselves namespaced — kinora rejects bare custom keys.)

**Rejected:** dotted namespaces (`jig.rust/slug`) and pure slash paths
(`jig/rust/slug`). Both work as bridges, but they introduce a *second*
separator convention that kinora would have to be taught to migrate. Reusing
kinora's existing `::` means the migration is mechanical and the ledger stays
internally consistent.

**Escape hatch:** none — this is a naming law, enforced by the `jig::meta::rule-schema`
rule. A kino that violates it is not a valid rule kino.
