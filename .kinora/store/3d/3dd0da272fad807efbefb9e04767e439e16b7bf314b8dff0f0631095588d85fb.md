# The dylint mechanism (deferred)

Some rules need what clippy can't express and an architectural test can't reach:
**type- and AST-directed** checks — "this trait may not be implemented for
foreign types", "no `impl` block over N methods", "this function's signature must
return a domain error, not `anyhow`". The mechanism for these is **dylint**:
custom lints written in Rust, run by `cargo dylint`, emitting rustc-style
diagnostics. On the graduation ladder they sit at rung 1 (warn) by default, rung
2 (deny) via dylint config — exactly like the clippy rules.

## Integration plan (for when it is built)

- **Toolchain.** A dylint lint links `rustc_private`, so the *lint crate* builds
  with a **nightly** toolchain carrying `rustc-dev` + `llvm-tools`, pinned in its
  own `rust-toolchain.toml`. This is independent of the workspace's stable
  toolchain — project code still builds on fenix stable; only the lint dylib uses
  nightly. fenix supplies the nightly (`complete`, or a pinned date).
- **Nix.** Add an opt-in `dylint = true` to the rust jig: it puts `cargo-dylint`
  + `dylint-link` + the pinned nightly on the devShell, and builds the lint dylib
  via crane with that nightly.
- **Diagnostics.** A bacon job `cargo dylint --all` feeds the existing
  `.bacon-claude-diagnostics` channel (dylint emits rustc JSON diagnostics, so
  the current `cargo_json` analyzer applies) — the agent reads violations the
  same way it reads clippy's.
- **Rule back-ref.** A dylint lint cites its rule kino in a `// rule:
  jig::rust::<slug>` comment beside `declare_lint!`, the same convention clippy
  lines and architectural tests use.

## Why deferred (not built now)

**Rejected: building the scaffold speculatively.** It would add a fragile,
nightly-pinned second toolchain to a currently clean stable-only flake, and there
is **no rule that needs it** — the one structural rule (`prefer-file-modules`) is
a source-walking architectural test, which is lighter and needs zero new infra.
The framework's mechanism menu already lists dylint as "later, only when a rule
needs real AST/type analysis." Standing it up before a concrete rule demands it
is exactly the kind of unused infrastructure YAGNI warns against, and the
toolchain-version coupling makes speculative scaffolding a maintenance liability.

**Escape hatch / trigger to build:** the first rule that genuinely requires type
or AST resolution. At that point this plan is the starting point, and
`jig::task::dylint-scaffold` tracks the work.
