# A lint rule is a kino

A jig lint rule is represented as a **kino in the `rules` root** of the
project's kinora ledger — not as a bare clippy line, a comment, or prose in
`CLAUDE.md`. The kino is the durable, versioned source of truth for the rule's
*statement*, its *rationale*, and its *position on the graduation ladder*
(see the graduation-ladder spec). Enforcement artifacts (a clippy line, an
architectural test, a dylint lint) are downstream projections that cite the
rule kino by name.

The reasoning: a rule is structurally identical to a spec. "One spec = one
atomic contract" applies verbatim — the only difference is that a rule
constrains *how code is shaped* rather than *what the product does*. Building a
second registry for rules would duplicate machinery kinora already provides:
stable ids, versioning, `kino://` cross-references, git-committed truth, and
`render`. So rules reuse it.

**Rejected:** a standalone `rules.toml` manifest independent of kinora. It would
re-implement versioning and rationale-tracking, and split the project's durable
intent across two stores. The whole value of the ledger is one home for
"why the code is the way it is" — a rule is exactly that.

**Consequence / coupling:** the rules framework requires the kinora jig to be
present. This is an accepted, deliberate coupling. A degraded `RULES.md`
fallback for kinora-less projects is explicitly out of scope for v1.

**Escape hatch:** a rule that is not yet worth formalizing stays as prose in
`CLAUDE.md` (ladder rung 0) until it earns a kino. Nothing forces every
convention to become a rule kino immediately.
