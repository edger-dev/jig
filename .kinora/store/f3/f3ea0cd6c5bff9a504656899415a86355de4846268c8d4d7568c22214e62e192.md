# The graduation ladder

Every rule occupies one **rung**, and the framework's central purpose is to move
rules *down* the ladder over time — because each rung down removes the rule from
the agent's context budget and makes compliance cheaper and more reliable.

| Rung | Mechanism | Cost to the agent | Reliability |
|------|-----------|-------------------|-------------|
| **0 · prose** | `CLAUDE.md` / ledger text | must hold it in context | low — depends on the agent |
| **1 · warn** | clippy config, dylint (warn), or an architectural test | reads it after the fact (diagnostics / `cargo test`) | high — deterministic, deferred |
| **2 · deny** | clippy / rustc / dylint at deny/forbid | zero — compile fails | total |
| **3 · structural** | types / trybuild — the wrong code is unrepresentable | zero — cannot be written | total, by construction |

A rule is *born* at rung 0 (a hint) and graduates downward as a mechanism to
enforce it is found. `unsafe_code = "forbid"` is a rule already at rung 2.

**Default target: warn-first (rung 1).** When a rule is first mechanized it lands
at rung 1, not rung 2. Because jig's rust `CLAUDE.md` already makes "zero
warnings before commit" the standard, a rung-1 rule is a *de-facto* guardrail —
it blocks commits — while staying recoverable during iteration. Promotion to
rung 2 (deny) is a deliberate, per-rule decision, taken once the rule has proven
it has no false positives worth tolerating.

**Rejected:** deny-by-default. Sending every newly-mechanized rule straight to a
hard compile failure maximizes pressure but makes the agent's iteration loop
brittle and punishes rules that still have rough edges. Warn-first with
deliberate promotion is the calmer, more trustworthy default.

**Graduation is a re-version.** Moving a rule between rungs is a kinora
re-version that bumps its `rule::rung` metadata. The enforcement artifact cites
the rule so `grep` answers "what enforces this?" and a metadata query answers
"which rules are still rung-0 prose?" — the drift question.

**Escape hatch:** a specific call site may opt out of a mechanized rule with the
mechanism's native suppression (`#[allow(...)]`, a test carve-out) plus a
one-line justification. The rule stays; the exception is local and visible.
